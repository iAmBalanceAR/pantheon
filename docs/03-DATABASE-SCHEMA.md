# Pantheon — Database Schema

## Overview

All data lives in Supabase (PostgreSQL). Row Level Security (RLS) is enabled on all user-facing tables. The service-role key is used in server-side API routes to bypass RLS where needed.

---

Run via `supabase/migrations/001_initial.sql`

### Table: `projects`

Primary entity. One row per user project.

| Column | Type | Notes |
|--------|------|-------|
| id | UUID PK | uuid_generate_v4() |
| owner_id | UUID FK → auth.users | cascade delete |
| name | TEXT | generated by controller |
| spec | TEXT | raw user input |
| spec_score | INTEGER (1–5) | complexity score from controller |
| resource_tier | TEXT | micro/small/medium/large/enterprise |
| status | TEXT | scoping/active/paused/reviewing/completed/failed |
| budget_tokens | INTEGER | default 50000 |
| budget_dollars | DECIMAL(10,2) | default 5.00 |
| tokens_used | INTEGER | running total |
| cost_used | DECIMAL(10,2) | running dollar total |
| stack | JSONB | tech stack determined by controller |
| metadata | JSONB | arbitrary extra data |
| created_at | TIMESTAMPTZ | |
| updated_at | TIMESTAMPTZ | auto-updated by trigger |
| report | JSONB | **Added in Migration 002** — ProjectReport object |

**Status flow:**
```
scoping → active → reviewing → completed
                 ↘ failed
                 ↘ paused (user-initiated, can resume)
```

### Table: `teams`

Supports recursive team structure. Root team has parent_team_id = NULL, depth = 0.

| Column | Type | Notes |
|--------|------|-------|
| id | UUID PK | |
| project_id | UUID FK → projects | |
| parent_team_id | UUID FK → teams | NULL for root |
| name | TEXT | |
| purpose | TEXT | |
| depth | INTEGER | 0 = root |
| spawned_by | UUID FK → agents | which agent created this sub-team |
| status | TEXT | active/dissolved |
| created_at | TIMESTAMPTZ | |

**Sub-team spawning**: when the Controller Agent decides a task needs a sub-team, it emits `<spawn_team>` JSON in its output. The executor parses this and creates a new team row + agents + tasks. Sub-teams have depth > 0 and parent_team_id set.

## Migration 002 — reports + user agent profiles

File: `supabase/migrations/002_agent_profiles_and_reports.sql`

Adds `projects.report` (JSONB) and **`user_agent_profiles`**: per-user `custom_context` per core `role`, appended into each skill’s `## User Context` section at runtime.

### Table: `user_agent_profiles` (Migration 002)

Allows users to append custom context to specific agent roles for all their projects.

| Column | Type | Notes |
|--------|------|-------|
| id | UUID PK | |
| user_id | UUID FK → auth.users | |
| role | TEXT | one of the 9 valid roles |
| custom_context | TEXT | appended to base prompt at runtime |
| created_at / updated_at | TIMESTAMPTZ | |

The base prompt is **immutable** in the repo — skills live in `src/lib/agents/skills/*.md`. Custom context is **injected** at runtime. One row per user per role (UNIQUE constraint).

## Migration 003 — project deliverable files

File: `supabase/migrations/003_project_files.sql`

Stores agent output in `<file path="…">` blocks as browsable rows after each task completes.

### Table: `project_files`

| Column | Type | Notes |
|--------|------|-------|
| id | UUID PK | |
| project_id | UUID FK → projects | cascade delete |
| task_id | UUID FK → tasks | nullable |
| agent_id | UUID FK → agents | nullable |
| path | TEXT | relative path, unique per project |
| content | TEXT | full file body |
| created_at / updated_at | TIMESTAMPTZ | |

Unique on `(project_id, path)` — later tasks **overwrite** the same path. RLS: same `project_owner` pattern as `tasks`.

**Product**: **Files** tab on each project; **Download all (.zip)** via `/api/projects/[id]/files/zip`.

**If the app errors with** `Could not find the table 'public.project_files' in the schema cache`:

1. Confirm the table exists: **SQL Editor** → `select * from information_schema.tables where table_name = 'project_files';`
2. If no row: run the full `003_project_files.sql` script (includes `NOTIFY pgrst, 'reload schema';` at the end).
3. If the table exists but the error persists, run only: `NOTIFY pgrst, 'reload schema';`  
   Or in the Dashboard: **Project Settings → API → Reload schema** (wording may vary by Supabase version).

## Migration 005 — installed Skills Library rows

File: `supabase/migrations/005_user_installed_skills.sql`

### Table: `user_installed_skills`

| Column | Type | Notes |
|--------|------|-------|
| id | UUID PK | |
| user_id | UUID FK → auth.users | cascade delete |
| library_id | TEXT | stable id matching `skills-library/{id}.md` |
| display_name, icon, category, description | TEXT | copied from catalog at install time |
| skill_content | TEXT | full markdown body; user-editable after install |

UNIQUE `(user_id, library_id)`. RLS: users only see their own rows (`own_skills` policy).

## RLS Notes

All tables have RLS enabled. The pattern:
- Users can only SELECT/INSERT/UPDATE/DELETE their own rows (auth.uid() = owner_id or user_id)
- Server-side API routes use the **service role key** (SUPABASE_SERVICE_KEY) to bypass RLS for cross-project operations like the Banker and Auditor

Never use the service role key in client-side code.
